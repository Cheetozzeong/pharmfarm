# 약품 재고 관리 및 반품 로직 개선 요구사항

## 목적

약국에서 보유 중인 약품의 재고를 QR 기반으로 등록하고, 시리얼 번호(SN)를 기준으로 입고처를 추적하여 반품 가능 도매처를 확인할 수 있도록 한다.

또한 처방전 분석 API 결과와 보유 재고를 연결하여 약품별 증감 정보를 표시하고, 기준 데이터 연결 실패 시에도 운영자가 수정하거나 임시 데이터를 생성해 업무 흐름이 끊기지 않도록 한다.

## 기준 데이터

### 1번 기준 데이터: 표준코드 기반 약품 마스터

QR에서 추출한 `pc` 값은 표준코드로 보고, 1번 기준 데이터에서 제품코드(보험코드)와 한글상품명을 찾는 데 사용한다.

master 필요 컬럼:

| 컬럼명           | 설명                       |
| ---------------- | -------------------------- |
| 한글상품명       | 약품 한글명                |
| 약품규격         | 약품 규격                  |
| 제품총수량       | 제품 단위 총수량           |
| 표준코드         | QR `pc`와 매칭되는 코드    |
| 제품코드(개정후) | 보험코드로 사용할 제품코드 |

1번 기준 데이터 CSV에는 더 많은 컬럼이 있을 수 있지만, 운영 master 테이블에는 위 핵심 컬럼만 정규 컬럼으로 저장한다. 나머지 원본 값은 필요 시 staging 또는 `raw_json`으로 보관한다.

실제 입력 파일:

| 파일                     | 형식 | 비고                                                                                                                                                  |
| ------------------------ | ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `docs/1번기준데이터.csv` | CSV  | 헤더가 위 전체 원본 컬럼과 일치한다. 운영 master에는 `한글상품명`, `약품규격`, `제품총수량`, `표준코드`, `제품코드(개정후)`만 정규 컬럼으로 저장한다. |

실제 CSV 헤더:

```text
한글상품명,업체명,약품규격,제품총수량,제형구분,포장형태,품목기준코드,품목허가일자,전문/일반,대표코드,표준코드,제품코드(개정후),일반명코드(성분명코드),비고,취소일자,양도양수적용(공고)일자,양도양수종료일자,일련번호생략여부,일련번호생략사유,ATC코드,특수관리약품구분,의약품판독장비구분
```

### 2번 기준 데이터: 보험코드/가격 마스터

1번 기준 데이터에서 얻은 제품코드(보험코드)를 기준으로 가격 정보를 조회한다.

master 필요 컬럼:

| 컬럼명         | 설명     |
| -------------- | -------- |
| 제품코드       | 보험코드 |
| 제품명         | 제품명   |
| 규격           | 규격     |
| 단위           | 단위     |
| 상한금액표금액 | 가격     |

2번 기준 데이터 CSV에도 더 많은 컬럼이 있을 수 있지만, 운영 master 테이블에는 위 핵심 컬럼만 정규 컬럼으로 저장한다. 나머지 원본 값은 필요 시 staging 또는 `raw_json`으로 보관한다.

실제 입력 파일:

| 파일                       | 형식 | 비고                                                                                                                                                                                |
| -------------------------- | ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `docs/2번 기준 데이터.csv` | CSV  | `주성분코드_동일제형` 헤더가 따옴표 안 줄바꿈을 포함할 수 있고, `상한금액표금액` 컬럼명과 가격 값에 공백/쉼표가 포함될 수 있다. import 시 컬럼명은 공백과 줄바꿈을 제거해 비교한다. |

실제 CSV 헤더:

```text
연번,투여,분류,식약분류,"주성분코드
_동일제형",주성분코드,주성분갯수,주성분명,제품코드,제품명,업체명,규격,단위, 상한금액표 금액,전일,비고
```

## 기준 데이터 업데이트 정책

1번 기준 데이터와 2번 기준 데이터는 CSV 최신본을 바로 운영 테이블에 반영하지 않는다. 업로드된 CSV 데이터는 먼저 staging 테이블에 적재하고, 기존 master 테이블과 DB에서 비교한 뒤 검증 성공 시 반영한다.

처리 흐름:

1. CSV 파일을 업로드한다.
2. 업로드 작업(import job)을 생성한다.
3. CSV 데이터를 기준 데이터별 staging 테이블에 bulk insert 한다.
4. staging 데이터의 필수값, 형식, 중복 가능성을 검증한다.
5. staging 테이블과 master 테이블을 DB JOIN으로 비교한다.
6. 신규, 수정, 비활성 대상 건수를 계산한다.
7. 검증에 실패하면 master 테이블은 변경하지 않는다.
8. 검증에 성공하면 master 테이블에 신규/수정 데이터를 반영한다.
9. 최신 CSV에 없는 기존 master 데이터는 삭제하지 않고 비활성 처리한다.

기준 데이터별 적용:

| 구분            | 처리 방식                                         |
| --------------- | ------------------------------------------------- |
| 1번 기준 데이터 | 전체 CSV 최신본을 staging에 적재 후 master와 비교 |
| 2번 기준 데이터 | 전체 CSV 최신본을 staging에 적재 후 master와 비교 |

컬럼 저장 원칙:

| 테이블   | 저장 범위                                                                   |
| -------- | --------------------------------------------------------------------------- |
| staging  | CSV 원본 컬럼을 최대한 보관하여 검증, 중복 제외 건수 집계, 운영 보정에 사용 |
| master   | 서비스 로직에 필요한 핵심 컬럼만 정규 컬럼으로 저장                         |
| raw_json | master 정규 컬럼에 없는 원본 값을 보관해야 할 때 사용                       |

비활성 처리 정책:

- 기준 데이터는 실제 삭제하지 않는다.
- 최신 CSV에 없는 데이터는 `active = false` 또는 `deleted_at`과 유사한 비활성 상태로 관리한다.
- 기존 재고, 입고 이력, 처방 차감 이력이 과거 기준 데이터를 참조할 수 있으므로 물리 삭제는 피한다.
- 기준 데이터 조회 API는 기본적으로 활성 데이터만 조회한다.
- `includeInactive=true` 옵션이 있으면 비활성 데이터도 함께 조회한다.

중복 처리 정책:

- 1번 기준 데이터의 `표준코드`, 2번 기준 데이터의 `제품코드`는 기본 매칭 키로 사용한다.
- 다만 원본 CSV에 중복 가능성이 있을 수 있으므로 DB 설계 단계에서는 중복을 무조건 불가능하게 막기보다 import job 단위로 중복을 탐지하고 검토할 수 있게 한다.
- 중복이 없는 데이터만 master에 자동 반영한다.
- 중복이 발견된 키의 데이터는 master에 자동 반영하지 않는다.
- 중복 데이터는 후보로 보류하지 않고 import 결과의 중복 제외 건수로만 기록한다.
- 중복 데이터는 사용자가 별도 선택해 반영하지 않는다.

## CSV Import 실행 방식

기준 데이터는 대량 데이터이므로 업로드 요청 안에서 모든 처리를 끝내지 않는다. 업로드 API는 import job을 만들고 `jobId`를 반환하며, 실제 CSV 파싱/검증/반영은 백그라운드에서 처리한다.

운영상 기준:

- 사용자는 DB에 직접 접근하지 않는다.
- 사용자는 프론트에서 CSV를 업로드한다.
- 백엔드는 업로드 파일을 받아 서버에 저장하고 import job을 생성한다.
- 백엔드는 업로드 파일을 streaming 방식으로 읽는다.
- staging 적재는 JPA 단건 저장이 아니라 JDBC batch insert 또는 DB bulk insert 방식으로 처리한다.
- master 반영은 staging 적재와 검증이 끝난 뒤 DB SQL로 일괄 처리한다.
- import 실패 시 기존 master 데이터는 변경하지 않는다.
- 30만 건 기준 데이터는 동기 처리하지 않는다.

동기 처리와 비동기 처리의 차이:

| 방식            | 설명                                            | 판단                           |
| --------------- | ----------------------------------------------- | ------------------------------ |
| 동기 처리       | 업로드 요청이 끝날 때까지 사용자가 대기         | 30만 건 기준 데이터에는 부적합 |
| 비동기 job 처리 | 업로드 후 `jobId`를 받고 상태 API로 진행률 확인 | 권장                           |

비동기 import 흐름:

1. 프론트가 CSV 파일을 업로드한다.
2. 백엔드는 파일을 저장하고 import job을 `PENDING` 상태로 생성한다.
3. 백엔드는 즉시 `jobId`를 응답한다.
4. 백그라운드 작업이 job 상태를 `RUNNING`으로 변경한다.
5. CSV 파일을 streaming 방식으로 읽는다.
6. 코드/날짜/금액 값을 문자열 및 표준 포맷으로 정규화한다.
7. 정규화된 데이터를 1,000~5,000건 단위 JDBC batch insert로 staging 테이블에 적재한다.
8. staging 데이터를 검증하고 master와 비교한다.
9. 중복 없는 데이터만 master에 신규/수정/비활성 데이터로 일괄 반영한다.
10. 중복 제외 건수가 있어도 정상 반영이 끝나면 job 상태를 `SUCCESS`로 변경한다.
11. 실패 시 master를 변경하지 않고 job 상태를 `FAILED`로 변경한다.
12. 프론트는 job 상태 조회 API로 처리 결과와 실패 사유를 확인한다.

사용하지 않는 방식:

| 방식                             | 제외 이유                                              |
| -------------------------------- | ------------------------------------------------------ |
| JPA `save` 단건 반복             | 30만 건 기준으로 너무 느리고 트랜잭션/메모리 부담이 큼 |
| 업로드 요청 안에서 전체 처리     | HTTP timeout, 브라우저 대기, 재시도 문제 발생 가능     |
| master 테이블 직접 insert/update | 검증 실패 시 운영 데이터가 일부만 변경될 수 있음       |
| 기존 master 전체 삭제 후 재생성  | 처리 중 조회 공백과 실패 시 복구 문제가 있음           |

운영 추천 구현:

| 단계         | 권장 구현                          |
| ------------ | ---------------------------------- |
| 파일 수신    | multipart upload 후 서버 파일 저장 |
| CSV 파싱     | streaming parser                   |
| staging 적재 | JDBC batch insert                  |
| 비교/반영    | DB JOIN 기반 SQL                   |
| 실패 처리    | master 변경 없이 job 실패 처리     |
| 사용자 확인  | job 상태 조회, 실패 조회           |

이 방식이 필요한 이유:

- 30만 건 CSV 처리는 요청 시간이 길어질 수 있다.
- 사용자가 브라우저를 닫아도 서버 작업은 계속 진행될 수 있다.
- 실패하더라도 기존 master 데이터는 유지된다.
- 처리 결과와 실패 행을 사용자가 나중에 확인할 수 있다.
- 운영 DB에는 검증이 끝난 데이터만 반영된다.
- 중복 데이터는 운영 master에 반영하지 않는다.

## 계정 및 인증

회원 계정은 약국에 소속된다. 한 약국에는 기본적으로 2개의 계정을 지급할 예정이며, 두 계정은 같은 약국의 재고, 입고/반품 추적 데이터, 바로팜 cookie, 구매 내역, 도매처 직접 등록 데이터를 함께 사용한다.

계정 지급 정책:

| 구분             | 설명                                       |
| ---------------- | ------------------------------------------ |
| 약국             | 재고와 구매 내역의 실제 소유 단위          |
| 계정             | 약국에 소속된 로그인 주체                  |
| 기본 지급 수     | 약국당 2개 계정                            |
| 데이터 분리 기준 | 계정 ID가 아니라 약국 ID                   |
| 제한 계정        | 2개 중 1개 계정은 추후 일부 기능 제한 예정 |

따라서 재고/구매내역/도매처/cookie 데이터의 `memberKey`는 계정 ID가 아니라 약국 ID 또는 약국 식별키를 사용한다. 로그인한 계정이 달라도 같은 약국 소속이면 같은 재고와 구매 내역을 조회한다.

현재 구현 상태:

- 현재 구현은 약국 테이블과 약국 소속 계정 테이블을 사용한다.
- 약국 계정 로그인, 토큰 재발급, 내 정보 조회, 계정명 수정, 비밀번호 변경 API가 구현되어 있다.
- 약국/계정 생성 API가 구현되어 있으며, 요청 본문에 관리자 비밀번호를 포함해야 한다.
- CMS 계정 생성 화면은 프론트에서 `root` 로그인 ID 계정만 접근할 수 있게 제한한다.
- JWT 인증 필터가 `/api/v1/pharmfarm/**` 요청의 Access Token을 검증하고 인증 컨텍스트를 구성한다.
- 로그인 후 JWT의 `pharmacyId`를 `memberKey`로 사용하여 재고/구매내역/쿠키/도매처 데이터를 약국 기준으로 저장한다.
- 로그인, 토큰 재발급, health, swagger 경로만 공개하고, 기준 데이터 import/수정/재매칭은 `ADMIN` 권한이 필요하다.
- 그 외 `/api/v1/pharmfarm/**` API는 인증된 계정만 호출할 수 있다.
- 일반 사용자용 회원가입은 제공하지 않는다. 운영자가 CMS root 계정으로 약국과 기본 계정 2개를 생성한다.

약국 테이블 주요 항목:

| 항목           | 설명                                                           |
| -------------- | -------------------------------------------------------------- |
| ID             | 내부 약국 식별자. 재고/구매내역/도매처/cookie 데이터 분리 기준 |
| 약국명         | 표시 및 관리용 약국 이름                                       |
| 대표자명       | 선택 입력                                                      |
| 연락처         | 선택 입력                                                      |
| 사업자등록번호 | 선택 입력. 필요 시 중복 검증                                   |
| 주소           | 선택 입력                                                      |
| 상태           | `ACTIVE`, `INACTIVE`, `LOCKED` 등                              |
| 생성일시       | 약국 생성 시각                                                 |
| 수정일시       | 마지막 수정 시각                                               |

계정 테이블 주요 항목:

| 항목               | 설명                                             |
| ------------------ | ------------------------------------------------ |
| ID                 | 내부 계정 식별자                                 |
| 약국 ID            | 계정이 소속된 약국 ID                            |
| 로그인 ID          | 계정이 로그인에 사용할 고유 ID                   |
| 비밀번호           | 단방향 해시로 저장. 평문 저장 금지               |
| 계정명             | 사용자 표시명                                    |
| 계정 유형          | `PRIMARY`, `LIMITED` 등                          |
| 권한               | `PHARMACY_OWNER`, `PHARMACY_LIMITED`, `ADMIN` 등 |
| 데이터 마스킹 정책 | 금액 등 특정 데이터 숨김 처리 여부               |
| 상태               | `ACTIVE`, `INACTIVE`, `LOCKED` 등                |
| 마지막 로그인 일시 | 마지막 로그인 시각                               |
| 생성일시           | 계정 생성 시각                                   |
| 수정일시           | 마지막 수정 시각                                 |

비밀번호 저장 정책:

| 항목          | 정책                                 |
| ------------- | ------------------------------------ |
| 저장 방식     | BCrypt 같은 단방향 해시 사용         |
| 평문 보관     | 금지                                 |
| 로그인 검증   | 입력 비밀번호와 저장된 해시를 비교   |
| 비밀번호 변경 | 기존 비밀번호 확인 후 새 해시로 교체 |

인증 방식:

| 항목          | 정책                                                                          |
| ------------- | ----------------------------------------------------------------------------- |
| Access Token  | JWT. API 인증에 사용                                                          |
| Refresh Token | Access Token 재발급에 사용                                                    |
| 전달 방식     | `Authorization: Bearer {accessToken}`                                         |
| 토큰 subject  | 계정 ID                                                                       |
| 토큰 claims   | 약국 ID, 계정 유형, 권한을 포함                                               |
| 프론트 저장   | 운영 환경에서는 보안 정책에 맞춰 HttpOnly cookie 또는 안전한 저장 방식을 선택 |

토큰 설계:

| 값              | 포함 위치          | 사용 목적                                  |
| --------------- | ------------------ | ------------------------------------------ |
| `accountId`     | subject 또는 claim | 실제 로그인 계정 식별, 작업자 이력 저장    |
| `pharmacyId`    | claim              | 재고/구매내역/쿠키/도매처 데이터 조회 기준 |
| `role`          | claim              | API 접근 권한 판단                         |
| `accountType`   | claim              | 주 계정/제한 계정 구분                     |
| `maskingPolicy` | claim 또는 DB 조회 | 금액 등 일부 데이터 마스킹 판단            |

효율성을 위해 Access Token에는 `accountId`, `pharmacyId`, `role`, `accountType`을 포함한다. 단, 제한 정책이 자주 바뀌거나 즉시 반영이 필요하면 API 요청 시 계정 상태와 마스킹 정책을 DB에서 한 번 더 조회한다.

로그인 후 약국 식별:

1. 프론트가 로그인 API를 호출한다.
2. 백엔드는 로그인 ID와 비밀번호를 검증한다.
3. 검증에 성공하면 Access Token과 Refresh Token을 발급한다.
4. 프론트는 이후 API 요청에 Access Token을 전달한다.
5. 백엔드는 JWT subject에서 계정 ID를 추출한다.
6. 백엔드는 계정에 연결된 약국 ID를 확인한다.
7. 재고/구매내역/도매처/cookie 관련 API는 약국 ID를 `memberKey`로 사용한다.
8. 같은 약국의 2개 계정은 같은 재고와 구매 내역을 본다.
9. 다른 약국의 계정은 다른 재고와 구매 내역을 본다.

권한 정책:

| 권한               | 설명                                                                          |
| ------------------ | ----------------------------------------------------------------------------- |
| `PHARMACY_OWNER`   | 약국 주 계정. 약국의 재고, 입고, 반품, 구매내역, cookie, 약국별 도매처를 관리 |
| `PHARMACY_LIMITED` | 약국 제한 계정. 같은 약국 데이터를 조회/사용하지만 추후 일부 기능 제한 예정   |
| `ADMIN`            | 운영자 계정. 기준 데이터 import, 기준 데이터 수정 가능                        |

제한 계정 정책:

| 항목           | 정책                                                                                     |
| -------------- | ---------------------------------------------------------------------------------------- |
| 기본 상태      | 주 계정과 동일한 약국 데이터를 공유                                                      |
| 제한 방식      | API 자체 차단보다 특정 데이터 마스킹 또는 일부 기능 차단 중심                            |
| 금액 제한 예시 | 가격, 예상 금액, 구매 금액 등 금액성 필드를 `0` 또는 숨김 값으로 응답                    |
| 제한 적용 시점 | 어떤 값을 언제 마스킹할지는 추후 운영 정책으로 결정                                      |
| 제한 가능 기능 | 금액 조회, 기준 데이터 수정, 바로팜 cookie 등록/교체, 구매내역 동기화, 수동 재고 조정 등 |
| 제한 관리      | 계정 유형, 권한, 마스킹 정책 값으로 제어                                                 |

API 접근 정책:

| API 구분                | 인증               | 권한                                                      |
| ----------------------- | ------------------ | --------------------------------------------------------- |
| 약국/계정 등록          | 불필요             | 관리자 비밀번호 필요. CMS 화면은 root 계정만 노출         |
| 로그인                  | 불필요             | 공개                                                      |
| 토큰 재발급             | Refresh Token 필요 | 공개 API이지만 refresh 검증 필요                          |
| 내 정보 조회            | 필요               | 본인                                                      |
| 재고/입고/반품          | 필요               | `PHARMACY_OWNER` 또는 `PHARMACY_LIMITED`                  |
| 바로팜 cookie/구매내역  | 필요               | 약국별 데이터. cookie 등록/교체 권한은 정책으로 제한 가능 |
| 약국별 도매처           | 필요               | API는 인증 계정 기준. CMS 관리 화면은 root 계정만 노출    |
| 기준 데이터 import/수정 | 필요               | `ADMIN`                                                   |
| 공통 도매처 관리        | API 미제공         | 초기 운영에서는 DB 직접 등록. 필요 시 추후 API 추가       |

계정 관련 API:

| Method | Path                                 | 설명                                        |
| ------ | ------------------------------------ | ------------------------------------------- |
| POST   | `/api/v1/pharmfarm/auth/pharmacies`  | 관리자 비밀번호로 약국과 기본 계정 2개 생성 |
| POST   | `/api/v1/pharmfarm/auth/login`       | 약국 로그인                                 |
| POST   | `/api/v1/pharmfarm/auth/reissue`     | Refresh Token으로 토큰 재발급               |
| GET    | `/api/v1/pharmfarm/auth/me`          | 현재 로그인한 계정과 소속 약국 정보 조회    |
| PATCH  | `/api/v1/pharmfarm/auth/me`          | 현재 로그인한 계정 정보 수정                |
| PATCH  | `/api/v1/pharmfarm/auth/me/password` | 비밀번호 변경                               |

초기 운영의 약국/계정 생성 방식:

- 일반 사용자 회원가입 API는 만들지 않는다.
- 운영자가 CMS root 계정으로 로그인한 뒤 계정 생성 화면에서 약국 생성 API를 호출해 약국 1개와 소속 계정 2개를 등록한다.
- 약국 생성 API 자체는 JWT 없이 호출 가능하지만 관리자 비밀번호가 맞아야 처리된다.
- 계정 비밀번호는 백엔드가 BCrypt 같은 단방향 해시로 저장한다.
- 요청의 첫 번째 계정은 `PRIMARY` / `PHARMACY_OWNER`, 두 번째 계정은 `LIMITED` / `PHARMACY_LIMITED`로 생성한다.

CMS 화면 권한 정책:

| 화면                   | 노출 기준                                            |
| ---------------------- | ---------------------------------------------------- |
| 대시보드               | 로그인한 CMS 계정                                    |
| 재고/초과 처방/처방전  | 로그인한 CMS 계정                                    |
| 구매 내역              | 로그인한 CMS 계정                                    |
| 기준 데이터            | `root` 로그인 ID 계정                                |
| 기준 데이터 Import     | `root` 로그인 ID 계정                                |
| 도매처 관리            | `root` 로그인 ID 계정                                |
| 계정 생성              | `root` 로그인 ID 계정                                |

프론트의 `root` 제한은 운영 화면 노출 정책이다. 백엔드는 기준 데이터 import/수정은 `ADMIN` 권한으로 보호하고, 약국별 도매처 API는 인증된 계정의 `memberKey` 범위 안에서 동작한다.

## 등록 데이터

### 1번 등록 데이터: 약국 보유 약품 재고

약국이 현재 보유한 약품 단위의 재고 데이터다. 로그인 계정 자체가 아니라 로그인 계정이 소속된 약국을 기준으로 재고를 식별한다.

계정 및 인증 정책은 `계정 및 인증` 섹션을 따른다. 현재 구현은 JWT에서 추출한 계정의 소속 약국 ID를 `memberKey`로 사용한다.

필수/주요 항목:

| 항목                  | 설명                                                          |
| --------------------- | ------------------------------------------------------------- |
| 약국                  | 로그인 계정이 소속된 약국                                     |
| 보험코드 타입         | 급여/비급여/임의코드 등 코드 타입                             |
| 보험코드              | 기준 데이터에서 찾은 제품코드                                 |
| 약 이름               | 약품명                                                        |
| 가격                  | 2번 기준 데이터의 상한금액표금액. 없으면 0                    |
| 개수                  | 현재 보유 수량. 입고 시 1번 기준 데이터의 제품총수량만큼 증가 |
| 제품총수량            | 1번 기준 데이터의 제품총수량                                  |
| 예상 금액             | `상한금액표금액 * 개수`                                       |
| 기준 데이터 연결 상태 | 정상/이름 매칭/임의 생성/미등록 등                            |
| 등록 계정 ID          | 해당 보유 약품 재고를 최초 생성한 계정                        |

### 2번 등록 데이터: 입고/반품 추적 데이터

QR 1장 또는 개별 제품 단위의 입고 및 반품 추적 데이터다.

필수/주요 항목:

| 항목               | 설명                                          |
| ------------------ | --------------------------------------------- |
| 약국               | 로그인 계정이 소속된 약국                     |
| 1번 등록 데이터    | 연결된 보유 약품 재고                         |
| PC                 | 표준코드. QR에서 추출한 `pc`                  |
| SN                 | 시리얼 번호. QR에서 추출한 `sn`               |
| LOT                | 제조번호. QR에서 추출한 `LOT`                 |
| EXP                | 만기일. QR에서 추출한 `EXP`                   |
| 보험코드           | 입고 시 확정된 보험코드                       |
| 2번 기준 데이터 ID | 선택된 가격 마스터. 임의 약품이면 비워둘 수 있음 |
| 약명 스냅샷        | 입고 시 확정된 약명                           |
| 가격 스냅샷        | 입고 시 확정된 가격. 임의 약품이면 0          |
| 제품총수량         | 입고 시점의 1번 기준 데이터 제품총수량        |
| 반품누적수량       | 해당 입고 건에서 이미 반품 처리된 수량        |
| 도매처 명          | 입고 도매처                                   |
| 등록 계정 ID       | 해당 입고/반품 추적 데이터를 최초 등록한 계정 |
| 입고일시           | 입고 처리 시각                                |
| 마지막 반품일시    | 마지막 반품 처리 시각                         |

식별 및 검증 기준:

| 구분             | 기준                                  |
| ---------------- | ------------------------------------- |
| 개별 제품 식별   | `PC + SN`                             |
| 반품 조회 기본값 | 로그인 계정이 소속된 약국 + `PC + SN` |
| 보조 검증값      | `LOT`, `EXP`                          |
| 권장 DB 제약     | `UNIQUE(PC, SN)`                      |

`SN`은 개별 제품 식별자로 사용할 수 있지만, 실제 운영에서는 `PC`와 함께 저장하고 조회하는 것이 안전하다. `LOT`, `EXP`는 제조번호와 만기일 검증, QR 파싱 오류 확인, 제조번호/유통기한 단위 정책 대응을 위해 함께 저장한다.

반품 가능 수량:

```text
반품가능수량 = 제품총수량 - 반품누적수량
```

반품 요청 수량은 `반품가능수량` 이하이어야 하며, 실제 차감 시에는 약국 보유 재고의 현재 개수도 함께 확인한다.

### 3번 등록 데이터: 약국별 구매 내역

기존에 이미 약국에 존재하던 약은 프로젝트 입고 API를 거치지 않았기 때문에 2번 등록 데이터에 `PC + SN` 이력이 없을 수 있다. 이 경우 반품 시 어느 구매처에서 온 약인지 확인할 수 없으므로, 약국별 구매 내역을 별도 데이터로 관리한다.

구매 내역은 바로팜 주문내역 API를 호출해 수집한다. 사용자는 바로팜 인증에 필요한 cookie 데이터를 백엔드에 등록하고, 백엔드는 로그인 계정이 소속된 약국과 cookie를 연결해 저장한다. 이후 주문내역 동기화 API를 호출하면 백엔드가 저장된 cookie로 바로팜 주문내역 API를 호출하고, 응답을 품목 단위로 파싱해 구매 내역 DB에 저장한다.

바로팜 주문내역 API:

```text
GET https://api-v2.baropharm.com/me/orders?start_date={startDate}&end_date={endDate}&per_page={perPage}&page={page}&over_the_counter=null
```

인증 실패 응답 예시:

```json
{
  "message": "자격 인증데이터(authentication credentials)가 제공되지 않았습니다.",
  "error_code": null,
  "errors": {
    "detail": "자격 인증데이터(authentication credentials)가 제공되지 않았습니다."
  }
}
```

cookie 관리 정책:

| 항목      | 정책                                                                   |
| --------- | ---------------------------------------------------------------------- |
| 저장 단위 | 로그인 계정이 소속된 약국                                              |
| 저장 방식 | DB 저장. 평문 저장 금지, 복호화 가능한 양방향 암호화 저장              |
| 사용 범위 | 바로팜 주문내역 조회 API 호출에만 사용                                 |
| 갱신 방식 | 사용자가 새 cookie를 등록하면 기존 cookie를 교체                       |
| 실패 처리 | 인증 실패 응답을 받으면 인증 실패 상태로 표시하고 cookie 재등록을 요청 |

프론트 입력 방식:

| 항목        | 처리                                                                                 |
| ----------- | ------------------------------------------------------------------------------------ |
| `sessionid` | 사용자가 값만 입력한다. `sessionid=` 접두어가 붙어 들어와도 프론트에서 정리한다.     |
| `csrftoken` | 사용자가 값만 입력한다. `csrftoken=` 접두어가 붙어 들어와도 프론트에서 정리한다.     |
| 저장 payload | 프론트가 `sessionid={값}; csrftoken={값};` 문자열로 조합해 기존 `cookie` 필드로 전송 |

프론트는 두 입력값을 조합해 `sessionid=...; csrftoken=...;` 형태로 전송한다. 백엔드에 저장하는 값에는 `Cookie:` 헤더명 접두어를 넣지 않는다. 백엔드는 복호화한 값을 바로팜 요청의 HTTP `Cookie` 헤더 값으로 사용한다.

주문 내역 조회는 최초 1회 동기화를 기본 전제로 하지만, 동기화 도중 cookie가 만료될 수 있다고 가정한다. 자동 갱신은 하지 않으며, 인증 실패 위치를 저장한 뒤 사용자가 cookie를 다시 등록하면 실패 지점부터 재개할 수 있게 한다.

cookie 암호화 설정:

| 설정           | 값                                     |
| -------------- | -------------------------------------- |
| properties key | `pharmfarm.baropharm.cookie-secret`    |
| 환경변수       | `PHARMFARM_BAROPHARM_COOKIE_SECRET`    |
| 기본값 예시    | `change-me-baropharm-cookie-secret-32` |
| 방식           | 복호화 가능한 양방향 암호화            |

실제 운영 환경에서는 기본값을 사용하지 않고 환경변수로 별도 secret을 주입한다. 백엔드는 cookie 원문을 로그나 API 응답에 노출하지 않는다.

바로팜 주문내역 응답 구조:

| 경로                                | 설명                  |
| ----------------------------------- | --------------------- |
| `total`                             | 전체 주문 수          |
| `per_page`                          | 페이지당 주문 수      |
| `current_page`                      | 현재 페이지           |
| `last_page`                         | 마지막 페이지         |
| `items[]`                           | 주문 목록             |
| `items[].whole_saler`               | 주문 판매처           |
| `items[].items[]`                   | 주문 품목 목록        |
| `items[].items[].inventory`         | 바로팜 재고/상품 정보 |
| `items[].items[].inventory.product` | 상품 상세 정보        |

구매 내역 저장 주요 항목:

| 항목              | 설명                                   |
| ----------------- | -------------------------------------- |
| 약국              | 로그인 계정이 소속된 약국              |
| source            | `BAROPHARM`                            |
| 외부 주문 ID      | 바로팜 `items[].id`                    |
| 외부 주문 품목 ID | 바로팜 `items[].items[].id`            |
| 거래일자          | `order_datetime`                       |
| 거래처            | `whole_saler.name`                     |
| 주문상품명        | `inventory_name` 또는 `inventory.name` |
| 상품명            | `inventory.product.name`               |
| raw_json          | 바로팜 주문 품목 원본 JSON             |

구매 내역 master 테이블은 반품 판매처 추정에 필요한 최소 컬럼만 정규 컬럼으로 저장한다. 거래처 후보 판단에 직접 사용하지 않는 값은 `raw_json`으로 보관한다.

구매 내역은 프로젝트 입고 데이터가 아니라 외부 구매 기록이다. 따라서 2번 등록 데이터와 분리해서 관리하며, 반품 도매처 확정 정보가 아닌 구매처 추정 후보를 제공하는 용도로 사용한다. 구매 내역 동기화는 약국 보유 재고를 증가시키거나 감소시키지 않는다.

구매처 추정 조회 기준:

1. 반품 조회 요청의 `PC`로 1번 기준 데이터를 조회한다.
2. 1번 기준 데이터에서 한글상품명을 얻는다.
3. 한글상품명, 주문상품명, 상품명은 공백을 제거한 정규화 값으로 비교한다.
4. 로그인 약국의 구매 내역에서 주문상품명 또는 상품명 중 하나라도 한글상품명과 겹치는 후보를 `LIKE`로 조회한다.
5. 구매 내역 후보가 여러 건이면 거래처별 후보 목록으로 반환한다.

## 도매처 관리 정책

도매처는 DB 테이블로 관리한다.

CMS의 도매처 관리 화면은 root 계정에서만 노출한다. 입고 모바일 화면과 API는 로그인한 약국의 도매처 선택을 위해 계속 사용한다.

관리 범위:

| 구분          | 설명                                       |
| ------------- | ------------------------------------------ |
| 공통 도매처   | 전체 약국이 공통으로 사용할 수 있는 도매처 |
| 약국별 도매처 | 특정 약국이 직접 입력해 사용하는 도매처    |

입고/반품 이력에는 도매처 ID와 도매처명을 함께 저장하는 것을 권장한다.

| 항목            | 설명                                                                       |
| --------------- | -------------------------------------------------------------------------- |
| 도매처 ID       | DB의 도매처 테이블 참조                                                    |
| 도매처명 스냅샷 | 입고 당시의 도매처명. 도매처명이 수정되어도 과거 이력 표시를 유지하기 위함 |

## 재고 수량 및 이력 정책

재고의 `개수`는 포장 수량이 아니라 실제 사용/차감 가능한 수량으로 관리한다. 입고 시에는 1번 기준 데이터의 `제품총수량`만큼 증가하고, 처방 차감 또는 반품 시에는 요청 수량만큼 감소한다.

수량 기준:

| 구분        | 처리                                             |
| ----------- | ------------------------------------------------ |
| 입고        | `제품총수량`만큼 재고 증가                       |
| 처방전 차감 | 처방전 분석 결과의 `totalQuantity`만큼 재고 감소 |
| 반품        | 반품 요청 수량만큼 재고 감소                     |
| 예상 금액   | `상한금액표금액 * 현재 재고 개수`                |

재고 수량이 변경되는 모든 작업은 재고 변경 이력으로 저장한다.

| 변경 유형             | 설명                         |
| --------------------- | ---------------------------- |
| `RECEIPT`             | QR 입고로 인한 증가          |
| `RETURN`              | 반품으로 인한 감소           |
| `PRESCRIPTION_DEDUCT` | 처방전 분석 결과로 인한 감소 |
| `MANUAL_ADJUST`       | 운영자 수동 조정             |

재고 변경 이력에는 변경 전 수량, 변경 수량, 변경 후 수량, 변경 사유, 관련 처방전/입고/반품 참조값, 처리자를 저장한다. 데이터 소유 기준은 약국 ID이고, 실제 작업자는 계정 ID로 별도 저장한다.

재고 변경 이력 주요 식별값:

| 항목         | 설명                                                        |
| ------------ | ----------------------------------------------------------- |
| 약국 ID      | 재고 데이터 소유 약국                                       |
| 처리 계정 ID | 실제 입고/반품/차감/수동조정을 수행한 계정                  |
| 변경 유형    | `RECEIPT`, `RETURN`, `PRESCRIPTION_DEDUCT`, `MANUAL_ADJUST` |
| 변경 전 수량 | 트랜잭션 안에서 잠근 뒤 확인한 기존 수량                    |
| 변경 수량    | 증가면 양수, 감소면 음수                                    |
| 변경 후 수량 | 변경 전 수량 + 변경 수량                                    |
| 참조 유형/ID | 입고 데이터, 처방전, 구매내역 등 관련 대상                  |

등록 계정과 처리 계정의 차이:

| 구분         | 설명                                                    |
| ------------ | ------------------------------------------------------- |
| 등록 계정 ID | 1번 등록 데이터 또는 2번 등록 데이터를 최초 생성한 계정 |
| 처리 계정 ID | 재고 변경 이력을 발생시킨 실제 작업 계정                |

예를 들어 A 계정이 QR 입고로 재고를 만들고, B 계정이 나중에 반품 처리하면 1번/2번 등록 데이터의 등록 계정 ID는 A이고, 반품 재고 변경 이력의 처리 계정 ID는 B가 된다.

반품은 별도 반품 이력 테이블을 만들지 않고 재고 변경 이력으로 추적한다. `PC + SN` 입고 추적 데이터에는 반품 가능 수량 계산을 위해 반품누적수량만 갱신한다.

반품 처리 시 재고 변경 이력 주요 항목:

| 항목            | 설명                                                                    |
| --------------- | ----------------------------------------------------------------------- |
| 2번 등록 데이터 | 반품 대상 입고/반품 추적 데이터. 구매 내역 후보 기반이면 비워둘 수 있음 |
| 반품 수량       | 해당 요청에서 반품한 수량                                               |
| 도매처 ID       | 반품 대상 도매처                                                        |
| 도매처명 스냅샷 | 반품 당시 표시할 도매처명                                               |
| 처리자          | 반품 처리 계정                                                          |
| 처리일시        | 반품 처리 시각                                                          |
| 메모            | 반품 사유 또는 참고 내용                                                |

## 기준 데이터 스냅샷 정책

1번/2번 기준 데이터는 CSV import로 계속 변경될 수 있으므로, 과거 입고/반품/재고 변경 이력은 당시 기준값을 스냅샷으로 보관한다.

스냅샷 저장 대상:

| 대상                                    | 저장할 값                                       |
| --------------------------------------- | ----------------------------------------------- |
| 입고/반품 추적 데이터                   | 약명, 보험코드, 가격, 제품총수량, 도매처명      |
| 재고 변경 이력                          | 약명, 보험코드, 가격, 변경 수량, 변경 후 수량   |
| 반품 재고 변경 이력                     | 약명, 보험코드, 제품총수량, 반품 수량, 도매처명 |
| 구매 내역 후보 기반 반품 재고 변경 이력 | 약명, 거래처, 거래일자, 반품 수량               |

기준 데이터가 수정되어도 과거 이력의 스냅샷 값은 자동 변경하지 않는다. 새 기준 데이터는 이후 입고/처방/반품 처리부터 적용한다.

## 동시성 처리 정책

재고 증가/차감 작업은 동시에 실행될 수 있으므로 트랜잭션 안에서 처리한다.

필수 처리:

| 상황        | 처리                                                                       |
| ----------- | -------------------------------------------------------------------------- |
| 입고        | 동일 약국/보험코드 재고 행을 잠근 뒤 수량 증가                             |
| 반품        | 재고 행과 `PC + SN` 입고 추적 행을 잠근 뒤 반품 가능 수량과 현재 재고 확인 |
| 처방전 차감 | 재고 행을 잠근 뒤 차감 가능 수량 확인 및 부족분 기록                      |
| 수동 조정   | 재고 행을 잠근 뒤 변경 이력 저장                                           |

차감 후 실제 재고 수량은 음수가 될 수 없다. 반품 가능 수량 초과, 이미 처리된 요청 재시도는 오류로 처리하고 변경 이력을 남기지 않는다. 처방전 차감은 재고가 부족해도 오류로만 끝내지 않고, 가능한 수량만 0까지 차감한 뒤 부족분을 초과 처방 내역으로 저장한다.

동시작업 충돌 해결 방식:

1. 재고를 변경하는 API는 반드시 DB 트랜잭션 안에서 실행한다.
2. 변경 대상 `pharmfarm_stock` 행을 `SELECT ... FOR UPDATE` 또는 JPA `PESSIMISTIC_WRITE`로 잠근다.
3. 반품처럼 `PC + SN` 입고 추적 데이터도 함께 바뀌는 경우 `pharmfarm_stock_item` 행도 같은 트랜잭션에서 잠근다.
4. 잠금을 잡은 뒤 현재 수량, 반품 가능 수량, 중복 처리 여부를 다시 조회한다.
5. 검증을 통과하면 수량을 변경하고 재고 변경 이력을 저장한다.
6. 검증 실패 시 수량과 이력을 저장하지 않고 오류를 반환한다.
7. 트랜잭션이 끝날 때 잠금이 해제된다.

현재 코드 반영 기준:

| 대상                  | 처리                                                                                                              |
| --------------------- | ----------------------------------------------------------------------------------------------------------------- |
| 재고 행               | `PharmfarmStockRepository.findForUpdate`, `findByMemberKeyAndInsuranceCodeForUpdate`에서 `PESSIMISTIC_WRITE` 사용 |
| 입고/반품 추적 행     | 반품 처리 시 `PharmfarmStockItemRepository.findByMemberKeyAndPcAndSnForUpdate`로 약국 + `PC + SN` 행 잠금         |
| 중복 입고             | `UNIQUE(PC, SN)`과 `saveAndFlush`로 DB 중복을 즉시 감지                                                           |
| 처방전 차감 행        | `PharmfarmPrescriptionStockDeductionRepository.findLineForUpdate`로 `prescriptionCode + lineNo` 행 잠금           |
| 처방전 차감 중복 생성 | DB unique key 충돌 시 기존 차감 행을 재조회해 사용                                                                |
| 수동 차감 실패 처리   | `findByIdAndMemberKeyForUpdate`로 실패 항목을 잠그고, 이미 처리된 항목은 재처리 차단                              |
| DB 중복 오류 응답     | `DataIntegrityViolationException`은 `409 Conflict`로 반환                                                         |

예시:

```text
현재 재고 10개

계정 A가 8개 반품 요청
계정 B가 5개 처방 차감 요청

1. 먼저 DB 잠금을 잡은 요청만 재고를 변경한다.
2. A가 먼저 처리되면 재고는 2개가 된다.
3. B는 A의 트랜잭션 종료 후 재고 2개를 다시 확인한다.
4. B의 5개 처방 중 2개만 차감하고 실제 재고는 0개가 된다.
5. 부족한 3개는 초과 처방 내역으로 저장하고 화면에서는 -3개 부족으로 표시한다.
6. 결과적으로 실제 재고가 -3개로 저장되는 상황은 막는다.
```

이 방식은 처리 순서를 DB가 직렬화하므로 같은 약국의 두 계정이 동시에 작업해도 재고 수량 정합성을 유지할 수 있다. 단, 잠금 범위는 변경 대상 재고 행으로 제한하여 전체 약국 재고가 불필요하게 막히지 않도록 한다.

## Import 결과 관리 정책

기준 데이터 import 작업은 결과를 import job으로 저장한다.

저장 항목:

| 항목           | 설명                                                                            |
| -------------- | ------------------------------------------------------------------------------- |
| 작업 ID        | import job 식별자                                                               |
| 데이터 구분    | 1번 기준 데이터 또는 2번 기준 데이터                                            |
| 상태           | `PENDING`, `RUNNING`, `SUCCESS`, `FAILED`, `AUTH_FAILED`, `PARTIAL_AUTH_FAILED` |
| 전체 행 수     | CSV 전체 행 수                                                                  |
| 처리 행 수     | 현재까지 파싱/적재한 행 수                                                      |
| 신규 건수      | master에 새로 추가된 건수                                                       |
| 수정 건수      | master에서 변경된 건수                                                          |
| 비활성 건수    | 최신 CSV에 없어 비활성 처리된 건수                                              |
| 중복 보류 건수 | 같은 키가 여러 건 있어 master 반영을 보류한 건수                                |
| 실패 건수      | 검증 또는 반영 실패 건수                                                        |
| 실패 상세      | 행 번호, 컬럼명, 실패 사유                                                      |
| 중복 제외 건수 | 같은 키가 여러 건 있어 master 반영에서 제외한 건수                              |
| 시작/종료 시각 | 처리 시간 추적                                                                  |
| 업로드 파일명  | 원본 파일명                                                                     |

import job이 실패하면 master 테이블은 변경하지 않는다. 실패 상세는 운영자가 CSV를 수정한 뒤 다시 업로드할 수 있도록 조회 가능해야 한다.

처리 행 수와 전체 행 수를 기준으로 프론트에서 진행률을 표시할 수 있다. 중복 데이터는 master에 반영하지 않고 결과 건수로만 표시한다.

구매 내역 동기화 job은 바로팜 API 호출 결과도 함께 저장한다.

| 항목                 | 설명                                         |
| -------------------- | -------------------------------------------- |
| 요청 시작일/종료일   | 바로팜 주문 조회 기간                        |
| 요청 perPage         | 바로팜 API 페이지당 요청 수. 기본값 500      |
| 전체 페이지 수       | 바로팜 응답 `last_page`                      |
| 처리 페이지 수       | 현재까지 호출 완료한 페이지                  |
| 조회 기간            | 사용자가 요청한 시작일/종료일                |
| 마지막 성공 page     | 인증 실패 또는 장애 발생 전 마지막 성공 page |
| 실패 page            | 인증 실패 또는 장애가 발생한 page            |
| 외부 API 오류 메시지 | 바로팜 인증 실패 또는 호출 실패 메시지       |
| 인증 실패 여부       | 인증 실패 판단값                             |

바로팜 주문내역 동기화 중 인증 실패 처리:

1. 바로팜 API 호출 중 인증 실패 응답을 받는다.
2. 백엔드는 추가 페이지 호출을 중단한다.
3. 시작부터 인증 실패면 job 상태를 `AUTH_FAILED`로 저장한다.
4. 일부 페이지 저장 후 인증 실패면 job 상태를 `PARTIAL_AUTH_FAILED`로 저장한다.
5. 조회 기간, 마지막 성공 page, 실패 page를 저장한다.
6. 프론트는 사용자에게 cookie 재등록을 안내한다.
7. 사용자가 cookie를 재등록한다.
8. 프론트가 동기화 재개 API를 호출한다.
9. 백엔드는 마지막 성공 다음 page부터 다시 조회한다.

## 에이전트 수집 반영 기준

약국 PC 에이전트는 EPharm 로컬 DB에서 현재 재고, 약품 마스터, 도매처, 처방 라인을 수집해 백엔드로 전송한다.

현재 구현 기준:

| 수집 대상        | 처리                                                                                                                                 |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| 재고 스냅샷      | `pharmfarm_agent_stock_snapshot`에 원본 수량을 저장한다. 0 또는 음수도 원본 스냅샷 관점에서는 저장 대상이다.                         |
| 실제 운영 재고   | `pharmfarm_stock` 반영은 수량이 1 이상일 때만 수행한다. 수량이 0 이하인 스냅샷은 실제 운영 재고 생성/갱신에서 제외한다.              |
| 약품 마스터      | 약국별 agent drug master에 저장하고, 기존 운영 재고명이 비어 있거나 보험코드명으로만 되어 있으면 약명 보정에 사용한다.              |
| 도매처           | 에이전트가 수집한 도매처명을 약국별 도매처 후보로 upsert한다.                                                                         |
| 처방 라인        | `prescriptionCode + lineNo` 기준으로 중복 차감을 방지하고, 처방 전체 상세 조회를 위해 agent prescription line에도 저장한다.           |
| 처방일시         | 에이전트는 `dosedAt`, `ps_dosed_at`, `ps_dosdate`를 로컬 타임존 offset이 포함된 ISO 문자열로 전송한다. 원본 문자열은 `ps_dosdate_raw`로 보존한다. |

대체 처방 처리:

| EPharm 원본값       | PharmFarm 해석              | 재고 처리                                                                 |
| ------------------- | --------------------------- | ------------------------------------------------------------------------- |
| `pd_extype = 0/null` | 일반 처방                   | 기존 처방 라인과 동일하게 재고 차감 및 부족분 판단 대상이다.             |
| `pd_extype = 1`      | 대체 전 원처방              | 처방 상세에는 보존하지만 재고 차감, 차감 실패, 초과 처방 생성 대상이 아니다. |
| `pd_extype = 2`      | 실제 대체 조제/대체 처방 행 | 실제 지급된 약품으로 보고 재고 차감 및 부족분 판단 대상이다.             |
| `pd_extype = 9`      | 저가 대체조제 가산 row      | 처방 상세에는 보존하지만 재고 차감, 차감 실패, 초과 처방 생성 대상이 아니다. |

에이전트는 `pd_extype`, `pd_exrow`, `pd_element` 원본값과 `substitutionRole` 해석값을 함께 전송한다. `pd_element`가 같고 수량/용법이 같은 `pd_extype=1` 행과 `pd_extype=2` 행은 같은 대체 처방 쌍으로 볼 수 있다. `pd_extype=9` 행은 같은 `pd_element` 그룹의 저가 대체조제 가산 row로 보고 `substitutionRole=SURCHARGE`, `affectsStock=false`로 전송한다. 서버는 처방 상세 조회 API에 세 행을 모두 내려주어 프론트에서 `대체 전 처방`/`대체 처방`/`저가 대체 가산`을 표시할 수 있게 한다.

금일 처방 재수집 테스트:

1. 에이전트의 `-ResyncTodayPrescriptions` 모드는 오늘 날짜의 EPharm 처방을 다시 조회해 `/agent/prescriptions`로 전송한다.
2. 이 payload는 `syncMode=TODAY_OVERWRITE`, `overwriteExisting=true`, `resyncRequestId`를 포함한다.
3. 백엔드는 같은 `pharmacyId + prescriptionCode`의 기존 agent prescription line을 새 payload 기준으로 대체 저장한다.
4. 같은 `prescriptionCode + lineNo` 차감 행이 이미 있고 약국 사용자가 상태 변경, 주문 처리, 수동 처리, 대체 약품 처리 등을 하지 않았다면 테스트 목적의 재계산을 허용할 수 있다.
5. 이미 운영 상태 변경이 들어간 처방은 무조건 덮어쓰지 말고 서버에서 거부하거나 별도 관리자 승인 흐름으로 분리한다.

수량 처리:

1. 재고 스냅샷의 0 또는 음수는 실제 재고 수량으로 바로 만들지 않는다.
2. 초과 처방 내역은 재고 스냅샷 수량이 음수라서 생성되는 것이 아니라, 처방 차감 요청 수량이 실제 보유 재고보다 클 때 생성된다.
3. 처방 수량이 현재 재고보다 많으면 실제 재고는 0까지만 차감하고 부족 수량을 `shortageQuantity`로 저장한다.
4. 처방 약품과 매칭되는 실제 재고가 없으면 초과 처방이 아니라 차감 실패 항목으로 저장하고 수동 처리를 기다린다.

## 입고 처리 흐름

1. 백엔드는 도매처 리스트를 제공한다.
2. 프론트는 사용자가 도매처를 선택하게 한다.
3. 프론트는 QR을 스캔하여 다음 값을 추출한다.
   - `pc`: 표준코드
   - `sn`: 시리얼 번호
   - `LOT`: 제조번호
   - `EXP`: 만기일
4. 프론트는 QR을 스캔할 때마다 `pc`만 백엔드의 QR 검증 API로 전송한다.
5. 백엔드는 `pc`를 사용해 1번 기준 데이터에서 표준코드를 조회한다.
6. 1번 기준 데이터에서 제품코드(보험코드), 한글상품명, 제품총수량을 얻는다.
7. 제품코드(보험코드)가 있으면 2번 기준 데이터의 제품코드를 정확히 조회한다.
8. 제품코드로 2번 기준 데이터를 찾으면 보험코드와 2번 기준 데이터 1건을 반환한다.
9. 제품코드로 찾지 못하면 1번 기준 데이터의 한글상품명과 2번 기준 데이터의 제품명을 공백 제거 정규화한 뒤 `LIKE` 조회하고, 조회된 2번 기준 데이터 후보 목록을 반환한다.
10. 이름으로도 찾지 못하면 보험코드는 비워두고 2번 기준 데이터 후보 목록은 빈 배열로 반환한다.
11. 프론트는 QR별 검증 결과를 사용자에게 보여주고, 사용자가 해당 QR을 입고 목록에 넣거나 제외할 수 있게 한다.
12. 2번 기준 데이터 후보가 여러 개면 프론트에서 사용자가 실제 입고할 약 정보를 선택한다.
13. 후보가 없거나 사용자가 2번 기준 데이터를 선택하지 않는 경우, 프론트는 등록할 임의 약품의 약명과 보험코드를 입력받는다.
14. 최종 입고 API는 도매처, QR 원본 정보(`pc`, `sn`, `lot`, `exp`)와 프론트에서 확정한 약 정보를 함께 받는다.
15. 선택된 2번 기준 데이터가 있으면 1번 기준 데이터와 선택된 2번 기준 데이터를 연결해 재고를 등록한다.
16. 선택된 2번 기준 데이터가 없으면 요청으로 받은 임의 약품명과 보험코드로 가격 0의 임의 재고를 등록한다.
17. 1번 등록 데이터의 재고 개수를 1번 기준 데이터의 제품총수량만큼 증가시킨다.
18. 2번 등록 데이터에 `PC`, `SN`, `LOT`, `EXP`, 제품총수량, 도매처 ID, 도매처 명, 확정 약 정보를 저장한다.

QR 검증 API 응답 기준:

| 상황                                                          | 응답                                                                         |
| ------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| 보험코드가 있고 2번 기준 데이터가 정확히 매칭됨               | `insuranceCode` 있음, `priceMasters` 1건                                     |
| 보험코드로 2번 기준 데이터를 찾지 못했고 이름으로 후보를 찾음 | `insuranceCode`는 1번 기준 데이터 값 또는 비어 있음, `priceMasters` 1건 이상 |
| 이름으로도 후보를 찾지 못함                                   | `insuranceCode` 비어 있음, `priceMasters` 빈 배열                            |

QR 검증 API는 재고를 증가시키거나 2번 등록 데이터를 만들지 않는다. 실제 재고 증가, 임의 약품 생성, 입고/반품 추적 데이터 저장은 최종 입고 API에서만 수행한다.

## 기준 데이터 매칭 규칙

QR `pc` 기준 매칭 순서:

1. `pc`로 1번 기준 데이터의 표준코드를 조회한다.
2. 1번 기준 데이터에서 제품코드(보험코드)와 한글상품명을 얻는다.
3. 제품코드(보험코드)로 2번 기준 데이터의 제품코드를 정확히 조회한다.
4. 2번 기준 데이터에 제품코드가 존재하면 해당 데이터를 우선 후보로 사용한다.
5. 제품코드로 찾지 못하면 1번 기준 데이터의 한글상품명과 2번 기준 데이터의 제품명을 공백 제거 정규화한 뒤 `LIKE` 조회한다.
6. 이름으로 찾은 2번 기준 데이터가 여러 건이면 후보 목록을 프론트에 반환하고 사용자가 선택하게 한다.
7. 이름으로도 찾지 못하거나 사용자가 2번 기준 데이터를 선택하지 않으면 최종 입고 시 임의 약품 등록 정보가 필요하다.
8. 임의 데이터의 가격은 0으로 설정한다.

임의 보험코드 정책:

1. 백엔드는 임의의 보험코드를 생성한다.
2. 임의 보험코드 형식은 `{약국ID}PF{일련번호}`로 한다.
3. 생성된 코드는 2번 기준 데이터의 제품코드와 중복되면 안 된다.
4. 생성된 코드는 같은 약국의 기존 임의 보험코드와 중복되면 안 된다.
5. 프론트에는 "해당 코드로 반영되었다"는 정보를 제공한다.
6. 사용자가 수정을 원하면 관리자 페이지 또는 수정 화면에서 임의 보험코드를 변경할 수 있게 한다.
7. 임의 보험코드를 실제 보험코드로 수정할 때 같은 약국에 해당 실제 보험코드 재고가 이미 있으면 기존 임의 재고를 실제 보험코드 재고로 병합한다.
8. 수정하려는 보험코드가 2번 기준 데이터의 활성 제품코드에 있으면 즉시 실제 재고 정보로 전환한다.
9. 수정하려는 보험코드가 2번 기준 데이터에 없으면 임의 재고 상태를 유지하고 보험코드만 수정한다. 가격과 약명은 기존 임의 재고 값을 유지한다.
10. 이후 2번 기준 데이터 import/update 시 해당 보험코드가 새로 들어오면 백엔드가 자동 재매칭하여 약명, 가격, 상태를 실제 재고로 갱신한다.
11. 같은 약국에 해당 실제 보험코드 재고가 이미 있으면 임의 재고의 수량을 실제 재고에 더하고, 기존 임의 재고는 삭제하지 않고 비활성 처리한다.
12. 보험코드 수정, 병합, 전환 작업은 재고 변경 이력에 `VIRTUAL_MERGE`로 저장한다.

보험코드 검증 및 생성 정책:

1. 프론트에서 임의 약품 보험코드를 직접 입력할 수 있으므로 보험코드 중복 확인 API를 제공한다.
2. 중복 확인 대상은 2번 기준 데이터의 활성 제품코드와 같은 약국의 활성 재고 보험코드다.
3. 백엔드는 임의 보험코드 랜덤 생성 API를 제공한다.
4. 랜덤 생성 API는 `{약국ID}PF{일련번호}` 형식으로 생성하고, 2번 기준 데이터와 같은 약국 재고에 중복되지 않는 값을 반환한다.
5. 최종 입고 API는 요청받은 임의 보험코드가 중복이면 입고를 거부한다.

비급여 또는 기준 데이터에서 보험코드를 확정할 수 없는 약품도 같은 임의 보험코드 정책을 따른다.

## 기준 데이터 수정 필요 기능

1번 기준 데이터와 2번 기준 데이터가 연결되지 않는 경우 운영자가 보정할 수 있어야 한다.

필요 기능:

| 기능                 | 설명                                        |
| -------------------- | ------------------------------------------- |
| 1번 기준 데이터 조회 | 표준코드, 제품코드, 약명 등으로 검색        |
| 1번 기준 데이터 수정 | 제품코드, 한글상품명, 제품총수량 등 보정    |
| 매칭 재시도          | 수정 후 2번 기준 데이터와 다시 연결         |
| 임의 데이터 표시     | 임의 보험코드/가격 0으로 생성된 데이터 구분 |

## 처방전 분석 API 연동

기존 처방전 분석 API는 보험코드로 약 종류를 찾는다.

현재 또는 추가로 활용 가능한 값:

| 항목        | 설명                |
| ----------- | ------------------- |
| 보험코드    | 처방 약품 코드      |
| 약명        | 처방 약품명         |
| 비급여 여부 | 급여/비급여 판단 값 |

처리 흐름:

1. 처방전 분석 API 결과에서 약명을 얻는다.
2. 약명으로 1번 기준 데이터를 조회한다.
3. 처방전 분석 API 결과의 `totalQuantity`를 지급되는 개수로 보고 차감 수량으로 사용한다.
4. 보유 재고와 연결되고 수량이 충분하면 해당 약품의 재고를 차감하고 차감 결과를 기록한다.
5. 보유 재고와 연결되었지만 수량이 부족하면 가능한 수량만 0까지 차감하고 부족분을 초과 처방 내역으로 저장한다.
6. 보유 재고를 찾지 못하면 미등록약 또는 임의 약품 후보로 표시한다.
7. 차감 처리되지 않은 항목은 별도 목록에서 관리한다.
8. 약국 관리자는 차감 실패 항목을 `선택한 재고에서 차감`, `임의 약품으로 처리`, `재고 반영 없이 완료` 중 하나로 선택해 처리한다.

비고:

- 처방전 연동 로직의 미등록/임의 데이터 판단은 확정값이 아니라 상태값으로 관리한다.
- 처방전 분석 결과로 재고를 차감하되, 어떤 처방전/약품/수량 기준으로 차감되었는지 이력을 남긴다.
- 차감 실패 항목은 재처리 또는 수동 확인할 수 있도록 별도 관리한다.
- 처방전 차감 실패 항목은 재고를 차감하지 않고 실패 항목으로 저장한 뒤 약국 관리자가 수동 처리한다.
- 실패 항목 상태는 `FAILED`, `RESOLVED`로 관리하고, 부족분이 발생한 항목은 `PARTIAL_DEDUCTED` 또는 `SHORTAGE`로 관리한다.
- 실제 재고 수량은 음수로 저장하지 않는다. 재고는 0까지 차감하고 부족분은 `shortageQuantity`와 화면용 `displayAfterQuantity`로 표시한다.
- 자동 차감은 처방전 약명과 재고 약명을 공백 제거 정규화한 뒤 양방향 포함 검색으로 먼저 비교한다.
- 처방전 보험코드는 비급여약에서 임의 값이 들어올 수 있으므로 보험코드만으로는 차감하지 않는다. 보험코드로 재고를 찾더라도 처방전 약명과 재고 약명이 함께 매칭될 때만 차감한다.
- 실제약 입고 재고명은 2번 기준 데이터 제품명을 사용하므로 처방전 약명과의 이름 기반 매칭률을 높인다.

처방전 차감 실패 수동 처리:

| 선택값               | 처리                                                                                                                        |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| 임의 약품으로 처리   | 약국별 임의 보험코드 정책에 따라 임의 약품을 생성하고 실패 항목의 차감 수량만큼 재고 변경 이력을 남긴다. 가격은 0으로 둔다. |
| 선택한 재고에서 차감 | 약국 관리자가 선택한 기존 재고 약으로 매칭하고 해당 재고를 차감한다.                                                        |
| 재고 반영 없이 완료  | 재고 차감 없이 실패 항목만 처리 완료로 변경한다. 실제 재고 수량은 변경하지 않는다.                                          |

처방전 초과 처방/부족분 처리:

| 항목           | 처리                                                               |
| -------------- | ------------------------------------------------------------------ |
| 처방 수량      | 처방전 분석 결과의 `totalQuantity`                                 |
| 실제 차감 수량 | 현재 재고에서 실제로 차감한 수량                                   |
| 부족 수량      | `처방 수량 - 실제 차감 수량`                                       |
| 실제 재고      | 0 미만으로 저장하지 않음                                           |
| 화면 표시      | 실제 차감 후 재고에서 부족 수량을 뺀 값으로 `-80` 같은 형태를 표시 |
| 부족분 상태    | `OPEN`, `ORDERED`, `HOLD`, `RESOLVED`, `IGNORED`                    |

초과 처방 상태값:

| 상태       | 화면 표시 | 처리 의미                                                           |
| ---------- | --------- | ------------------------------------------------------------------- |
| `OPEN`     | 주문 필요 | 기본 상태. 아직 주문 또는 보류 결정을 하지 않은 부족분              |
| `ORDERED`  | 주문 완료 | 부족분에 대해 주문을 넣은 상태. 실제 재고는 입고 처리 시 증가한다.  |
| `HOLD`     | 보류      | 당장 처리 목록에서 제외하고 별도 탭에서 확인할 부족분               |
| `RESOLVED` | 보류      | 과거 호환 상태. 프론트에서는 보류 그룹으로 표시                     |
| `IGNORED`  | 보류      | 과거 호환 상태. 프론트에서는 보류 그룹으로 표시                     |

초과 처방 상태 변경은 주문 진행 상태만 바꾸며 실제 재고 수량은 변경하지 않는다. 주문 완료 후 실제 재고가 들어오면 별도 입고/재고 조정 흐름으로 재고가 증가한다.

CMS 초과 처방 화면은 `주문 필요`, `주문 완료`, `보류` 탭을 상태별 리스트 필터로 사용한다. 탭을 바꾸는 것만으로 상세 화면으로 이동하지 않으며, 사용자가 리스트 행을 선택했을 때만 해당 항목 상세와 관련 처방전 전체 품목을 보여준다. 한 번 상태를 지정한 항목도 삭제하지 않고 다시 상태를 변경할 수 있어야 한다.

처방전 차감 실패 항목은 약국 관리자가 수동 처리 API로 처리한다. 처리 결과는 차감 실패 항목 상태와 재고 변경 이력에 반영한다. 초과 처방 항목은 리스트에서 행을 클릭하면 관련 처방전 전체 품목을 처방전형 상세 UI로 확인할 수 있다.

처방전 차감 중복 방지:

| 항목                  | 기준                                                      |
| --------------------- | --------------------------------------------------------- |
| 처방전 식별값         | `prescriptionCode`                                        |
| 처방 약품 라인 식별값 | `drugs.pd_no`                                             |
| 보험코드              | `drugs.pd_iscode`                                         |
| 차감 수량             | `drugs.pd_amount` 또는 기존 정규화 결과의 `totalQuantity` |

같은 `prescriptionCode + pd_no` 조합은 한 번만 차감한다. `pd_no`가 없으면 `prescriptionCode + pd_iscode + drug_name + pd_amount` 조합으로 중복 차감을 방지한다.

## 반품 처리 흐름

1. 프론트는 QR을 스캔하여 `pc`, `sn`, `lot`, `exp`를 백엔드로 전송한다.
2. 백엔드는 2번 등록 데이터에서 `pc + sn`으로 입고 이력을 조회한다.
3. `lot`, `exp`가 전달되면 저장된 값과 일치하는지 검증한다.
4. 2번 등록 데이터 조회 및 검증에 성공하면 도매처 명과 반품 가능 수량을 반환한다. 이 경우 도매처는 확정 정보다.
5. `pc + sn`으로 2번 등록 데이터를 찾지 못하면, `pc`로 1번 기준 데이터를 조회한다.
6. 1번 기준 데이터에서 한글상품명을 얻고, 같은 약국의 구매 내역 테이블에서 주문상품명 또는 상품명 기준으로 판매처 후보를 조회한다.
7. 구매 내역 후보가 있으면 판매처 후보 목록을 반환한다. 이 경우 판매처는 확정 정보가 아니라 추정 정보다.
8. 프론트는 사용자에게 "입고 이력은 없지만 구매 내역 기준으로 다음 판매처들이 추정된다"는 방식으로 안내한다.
9. 사용자가 구매처 후보를 확인하면 기존 6단계 확인 과정을 1~5개 후보 확인으로 줄일 수 있다.
10. 프론트가 반품 수량을 입력하고 반품하기 API를 호출한다.
11. 백엔드는 2번 등록 데이터 기반 반품인지, 구매 내역 후보 기반 반품인지 구분해 처리한다.
12. 2번 등록 데이터 기반 반품이면 다시 `pc + sn`으로 조회하고 `lot`, `exp`를 검증한다.
13. 2번 등록 데이터 기반 반품이면 반품 수량이 `제품총수량 - 반품누적수량`보다 작거나 같은지 검증한다.
14. 구매 내역 후보 기반 반품이면 선택한 구매 내역을 재고 변경 이력의 참고값으로 저장한다.
15. 약국 보유 재고의 현재 개수가 반품 수량 이상인지 검증한다.
16. 재고 변경 이력을 저장한다.
17. 연결된 1번 등록 데이터의 개수를 반품 수량만큼 차감한다.

반품은 해당 약이 어느 도매처에서 온 것인지 확인하고 해당 수량만큼 재고를 차감하기 위한 흐름이다. 따라서 상태값 중심으로 관리하기보다 `제품총수량`, `반품누적수량`, `반품가능수량`, 재고 변경 이력을 기준으로 관리한다.

반품 조회 결과 구분:

| 구분             | 설명                                                                                                                                  |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 확정             | 2번 등록 데이터에서 `PC + SN`으로 입고 이력을 찾은 경우                                                                               |
| 추정             | 2번 등록 데이터는 없지만 `PC`로 찾은 1번 기준 데이터의 한글상품명이 구매 내역의 주문상품명 또는 상품명과 겹쳐 판매처 후보를 찾은 경우 |
| 판매처 후보 없음 | 2번 등록 데이터와 구매 내역 모두에서 판매처 후보를 찾지 못한 경우                                                                     |

판매처 후보가 없는 경우는 우리 서비스에 입력된 구매/입고 근거가 없는 상태이므로 반품 처리 대상으로 등록하지 않는다.

## API 목록

### 계정 및 인증

| Method | Path                                 | 설명                                        |
| ------ | ------------------------------------ | ------------------------------------------- |
| POST   | `/api/v1/pharmfarm/auth/pharmacies`  | 관리자 비밀번호로 약국과 기본 계정 2개 생성 |
| POST   | `/api/v1/pharmfarm/auth/login`       | 약국 로그인                                 |
| POST   | `/api/v1/pharmfarm/auth/reissue`     | Refresh Token으로 토큰 재발급               |
| GET    | `/api/v1/pharmfarm/auth/me`          | 현재 로그인한 계정과 소속 약국 정보 조회    |
| PATCH  | `/api/v1/pharmfarm/auth/me`          | 현재 로그인한 계정 정보 수정                |
| PATCH  | `/api/v1/pharmfarm/auth/me/password` | 비밀번호 변경                               |

### 대시보드

| Method | Path                               | 설명                                                                                     |
| ------ | ---------------------------------- | ---------------------------------------------------------------------------------------- |
| GET    | `/api/v1/pharmfarm/dashboard`      | 오늘 처리할 일, 재고 요약, 오늘 재고 증감, 최근 활동, 연동 상태를 로그인 약국 기준으로 조회 |

CMS 대시보드는 API 응답 중 개발자용 연동 상태를 화면 전면에 노출하지 않고, 처방 실패/초과 처방/임의 재고/재고 0개 등 약사가 처리해야 할 일을 우선 표시한다. 금액성 데이터는 주 계정 또는 권한이 있는 계정에서만 보여준다.

### 도매처

| Method | Path                                              | 설명                                                                                 |
| ------ | ------------------------------------------------- | ------------------------------------------------------------------------------------ |
| GET    | `/api/v1/pharmfarm/wholesalers?keyword={keyword}` | 도매처 리스트 조회. `keyword`는 선택값이며 공백 제거 정규화 기준으로 도매처명을 검색 |
| POST   | `/api/v1/pharmfarm/wholesalers`                   | 약국별 도매처 직접 등록                                                              |
| PATCH  | `/api/v1/pharmfarm/wholesalers/{wholesalerId}`    | 약국별 도매처 수정                                                                   |

도매처 리스트는 공통 도매처와 로그인 약국이 직접 등록한 도매처를 함께 반환한다.


```json
{
  "wholesalerId": 10,
  "wholesalerName": "A도매",
  "items": [

      "sn": "SN123456",
      "lot": "LOT202606",
      "exp": "2027-12-31",
      "drugMasterId": 100,
      "insuranceCode": "643102120",
      "priceMasterId": 200,
      "virtualDrugName": null,
      "virtualInsuranceCode": null
    },
    {
      "pc": "8800000000001",
      "sn": "SN123457",
      "lot": "LOT202606",
      "exp": "2027-12-31",
      "drugMasterId": 101,
      "insuranceCode": null,
      "priceMasterId": null,
      "virtualDrugName": "기준 데이터 미매칭 약품",
      "virtualInsuranceCode": "2PF000001"
    }
```

입고 요청 약 정보 처리:

1. `priceMasterId`가 있으면 선택된 2번 기준 데이터를 기준으로 실제 재고를 등록한다.
2. 실제 재고의 약명은 선택된 2번 기준 데이터의 `productName`을 사용한다. 처방전 데이터가 2번 기준 데이터명과 유사하게 들어오므로 이름 기반 재고 차감 검색과 맞추기 위함이다.
3. 실제 재고의 보험코드는 선택된 2번 기준 데이터의 `productCode`, 가격은 `maxPrice`를 사용한다.
4. `priceMasterId`가 없으면 `virtualDrugName`, `virtualInsuranceCode`가 필수다.
5. `virtualInsuranceCode`는 2번 기준 데이터의 활성 제품코드와 같은 약국의 활성 재고 보험코드와 중복되면 안 된다.
6. 최종 입고 시에는 QR 검증 결과를 신뢰하지 않고 `pc`, `drugMasterId`, `priceMasterId`, 보험코드 중복 여부를 다시 검증한다.

### 재고

| Method | Path                                               | 설명                                                                 |
| ------ | -------------------------------------------------- | -------------------------------------------------------------------- |
| GET    | `/api/v1/pharmfarm/stocks?keyword={keyword}`       | 약품명, 정규화 약품명, 보험코드 기준 서버 검색. `controlledOnly` 지원 |
| GET    | `/api/v1/pharmfarm/stocks/{stockId}`               | 보유 재고 상세 조회                                                  |
| POST   | `/api/v1/pharmfarm/stocks`                         | 보험코드, 약품명, 가격, 초기 수량을 직접 입력해 재고 생성             |
| POST   | `/api/v1/pharmfarm/stocks/{stockId}/adjustments`   | 운영자 수동 재고 조정                                                |
| PATCH  | `/api/v1/pharmfarm/stocks/{stockId}/price`         | 보유 재고 가격 직접 수정                                             |
| POST   | `/api/v1/pharmfarm/stocks/{stockId}/merge-virtual` | 임의 재고를 실제 보험코드 재고로 전환 또는 병합                      |

수동 재고 조정은 증가/감소를 모두 허용한다. 조정 사유는 선택 입력값으로 둔다. CMS 재고 화면은 서버 검색을 사용하며 검색어는 2글자 이상일 때 실행한다. 전체/향정/비향정/임의 재고는 탭 또는 필터로 구분한다.

재고 직접 생성 요청 예시:

```json
{
  "stockTag": "VIRTUAL",
  "insuranceCode": "2PF000001",
  "drugName": "임의 등록 약품",
  "price": 0,
  "quantity": 10,
  "productTotalQuantity": 0,
  "memo": "초기 운영 수기 등록"
}
```

`stockTag`는 `EXISTING` 또는 `VIRTUAL`을 사용한다. 임의 재고는 기준 데이터로 가격을 자동 매칭할 수 없으므로 CMS에서 가격을 직접 입력하거나 이후 실제 보험코드로 병합한다.

재고 가격 수정 요청 예시:

```json
{
  "price": 120,
  "memo": "임의 재고 단가 보정"
}
```

임의 재고 병합 요청 예시:

```json
{
  "insuranceCode": "643102120",
  "memo": "기준 데이터 갱신 후 실제 보험코드로 병합"
}
```

임의 재고 병합 처리:

1. 로그인 계정이 소속된 약국의 활성 임의 재고인지 확인한다.
2. 같은 약국에 해당 보험코드의 활성 재고가 있으면 임의 재고 수량을 실제 재고에 더하고 기존 임의 재고는 비활성 처리한다.
3. 같은 약국에 해당 보험코드의 활성 재고가 없고 2번 기준 데이터에 보험코드가 있으면 기존 임의 재고를 실제 보험코드 재고로 전환한다.
4. 같은 약국에 해당 보험코드의 활성 재고가 없고 2번 기준 데이터에도 보험코드가 없으면 임의 재고 상태를 유지한 채 보험코드만 수정한다.
5. 이후 2번 기준 데이터 import/update 시 해당 보험코드가 들어오면 자동 재매칭하여 실제 재고로 전환한다.
6. 병합, 전환, 보험코드 수정 결과는 재고 변경 이력에 `VIRTUAL_MERGE`로 저장한다.

바로팜 cookie 등록 요청 예시:

```json
{
  "cookie": "sessionid=...; csrftoken=...;"
}
```

프론트에서는 `sessionid`와 `csrftoken` 값을 별도 입력받아 위 문자열로 조합한다. 사용자가 `Cookie:` 접두어 또는 `sessionid=`/`csrftoken=` 형태로 붙여 넣어도 프론트에서 값만 정리해 전송한다.

구매 내역 동기화 요청 예시:

```json
{
  "startDate": "2022-07-25",
  "endDate": "2025-07-25"
}
```

구매 내역 동기화 처리:

1. 로그인 계정이 소속된 약국의 바로팜 cookie를 조회한다.
2. cookie가 없으면 동기화를 시작하지 않고 인증 정보 필요 오류를 반환한다.
3. 프론트가 요청한 전체 기간을 그대로 사용한다. 월 단위로 기간을 나누지 않는다.
4. 바로팜 주문내역 API는 `per_page=500`을 기본값으로 호출한다.
5. 요청 기간에서 1페이지부터 `last_page`까지 page만 증가시키며 호출한다.
6. 바로팜 API 호출 실패 시 per_page를 낮춰 재시도하지 않고 job을 실패 처리한다.
7. 인증 실패 응답이면 시작 전 실패는 `AUTH_FAILED`, 일부 처리 후 실패는 `PARTIAL_AUTH_FAILED`로 저장한다.
8. 주문의 `items[].items[]`를 품목 단위 구매 내역으로 파싱한다.
9. 외부 주문 품목 ID 기준으로 중복 저장을 방지하고 신규/수정 반영한다.
10. 구매 내역 동기화는 판매처 추정용 데이터만 저장하며 약국 보유 재고는 변경하지 않는다.
11. 약국당 동시에 실행 가능한 바로팜 주문내역 동기화 job은 1개로 제한한다.

서버 재시작 처리:

1. 서버 시작 시 `RUNNING` 상태의 바로팜 주문내역 동기화 job을 조회한다.
2. 마지막 성공 page를 확인한다.
3. 해당 job을 마지막 성공 다음 page부터 재개한다.
4. 이미 저장된 주문 품목은 외부 주문 품목 ID 기준으로 중복 저장하지 않는다.

동기화 재개 처리:

1. `AUTH_FAILED` 또는 `PARTIAL_AUTH_FAILED` 상태의 job만 재개할 수 있다.
2. 재개 전 로그인 계정이 소속된 약국의 바로팜 cookie가 다시 등록되어 있어야 한다.
3. `PARTIAL_AUTH_FAILED`는 마지막 성공 page 다음 page부터 다시 호출한다.
4. `AUTH_FAILED`는 첫 처리 구간의 1페이지부터 다시 호출한다.
5. 이미 저장된 주문 품목은 외부 주문 품목 ID 기준으로 중복 저장하지 않는다.

처방전 차감 실패 항목 수동 처리 요청 예시:

```json
{
  "resolutionType": "EXISTING_STOCK",
  "stockId": 10,
  "memo": "관리자 수동 매칭"
}
```

`resolutionType` 값:

| 값                  | 설명                            |
| ------------------- | ------------------------------- |
| `VIRTUAL_DRUG`      | 임의 약품으로 처리              |
| `EXISTING_STOCK`    | 재고가 있는 약으로 매칭 후 차감 |
| `UNREGISTERED_DRUG` | 재고 반영 없이 완료             |

처방전 차감 실패 항목 수동 처리 규칙:

| resolutionType      | 필수 값   | 처리                                                                                                               |
| ------------------- | --------- | ------------------------------------------------------------------------------------------------------------------ |
| `VIRTUAL_DRUG`      | 없음      | 실패 항목의 약명으로 임의 약품을 생성한다. 가격은 0으로 두고, 실패 항목의 차감 수량만큼 재고 차감 이력을 저장한다. |
| `EXISTING_STOCK`    | `stockId` | 선택한 기존 재고 약으로 매칭하고 실패 항목의 차감 수량만큼 재고 차감                                               |
| `UNREGISTERED_DRUG` | 없음      | 재고 차감 없이 실패 항목만 처리 완료로 변경                                                                        |

초과 처방 상태 변경 요청 예시:

```json
{
  "shortageStatus": "HOLD",
  "memo": "주문하지 않고 보류"
}
```

`shortageStatus`는 `OPEN`, `ORDERED`, `HOLD`를 기본으로 사용한다. 백엔드는 호환을 위해 `ORDER_NEEDED`, `ORDER_COMPLETE`, `ON_HOLD`, `주문필요`, `주문완료`, `보류` 같은 별칭도 허용한다.

처리 상태:

| 상태               | 설명                                                    |
| ------------------ | ------------------------------------------------------- |
| `FAILED`           | 자동 차감 실패. 수동 처리 대기                          |
| `PARTIAL_DEDUCTED` | 재고 일부를 차감하고 부족분 처리 대기                   |
| `SHORTAGE`         | 매칭 재고는 있으나 실제 차감 수량 없이 부족분 처리 대기 |
| `RESOLVED`         | 수동 처리 완료. 처리 방식은 `resolutionType`으로 구분   |

`EXISTING_STOCK` 또는 `VIRTUAL_DRUG` 처리로 재고가 차감되면 재고 변경 이력에 `PRESCRIPTION_DEDUCT`로 저장한다. `UNREGISTERED_DRUG`는 재고를 차감하지 않고 실패 항목만 `RESOLVED`로 변경한다. 같은 실패 항목은 한 번만 수동 처리할 수 있으며, 이미 처리된 항목에 대한 중복 요청은 오류로 반환한다.

## 예외 및 상태 관리

| 상황                                                            | 처리                                                                                               |
| --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| QR 검증에서 `pc`로 1번 기준 데이터 미조회                       | 보험코드 없음, 2번 기준 데이터 후보 빈 배열 반환                                                   |
| QR 검증에서 2번 기준 데이터 미조회                              | 이름 `LIKE` 조회 시도                                                                              |
| QR 검증에서 이름 조회도 실패                                    | 보험코드 없음, 2번 기준 데이터 후보 빈 배열 반환                                                   |
| 최종 입고에서 2번 기준 데이터를 선택하지 않음                   | 임의 약품명과 임의 보험코드가 필수                                                                 |
| 최종 입고에서 임의 보험코드 중복                                | 중복 오류 반환                                                                                     |
| 최종 입고에서 `pc`, `drugMasterId`, `priceMasterId` 연결 불일치 | 기준 데이터 검증 오류 반환                                                                         |
| `pc + sn` 중복 입고                                             | 중복 오류 반환                                                                                     |
| `pc + sn` 동시 중복 입고                                        | DB unique key로 최종 방어하고 `409 Conflict` 반환                                                  |
| `pc + sn` 반품 조회 실패                                        | `pc`로 1번 기준 데이터를 찾고 한글상품명과 구매 내역의 주문상품명/상품명 기준으로 판매처 후보 조회 |
| `pc + sn` 반품 조회 실패 및 구매 내역 후보 없음                 | "판매처 후보를 찾지 못했습니다." 반환                                                              |
| `pc + sn`은 일치하지만 `lot` 또는 `exp` 불일치                  | QR 정보와 입고 이력이 다름 오류 반환                                                               |
| 반품 수량이 반품 가능 수량보다 큼                               | 반품 가능 수량 초과 오류 반환                                                                      |
| 재고 수량이 0인데 반품 요청                                     | 차감 불가 오류 반환                                                                                |
| 재고 수량이 반품 수량보다 적음                                  | 차감 불가 오류 반환                                                                                |
| 같은 처방전 라인 동시 차감                                      | `prescriptionCode + lineNo` 차감 행 잠금 및 unique key로 중복 차감 방지                            |
| 이미 처리된 처방전 차감 실패 항목 재처리                        | 오류 반환                                                                                          |
